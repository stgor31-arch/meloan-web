ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ) 
Проект 

PWA-приложение для учета и управления частными займами с возможностью последующей миграции в нативное iOS-приложение.
1. Цель проекта
Создать инструмент для частного кредитора (мастер-аккаунт), позволяющий:
•	учитывать выданные займы,
•	контролировать платежи и просрочки,
•	вести учет движения полученных средств.
Приложение также предоставляет личный кабинет заемщика для дисциплины платежей и прозрачности расчетов.
Проект разрабатывается по стратегии «Инструмент → Массовый продукт».
________________________________________
2. Платформа
•	Тип: Progressive Web App (PWA)
•	Поддержка мобильных браузеров (iOS / Android)
•	Возможность установки на главный экран
•	Backend с REST / GraphQL API
•	Архитектура, совместимая с будущим iOS-приложением
________________________________________
3. Роли пользователей
3.1 Мастер (кредитор)
•	Создает займы
•	Видит всех своих заемщиков
•	Подтверждает платежи
•	Видит аналитику и просрочки
•	Ведет учет распределения средств
3.2 Заемщик
•	Принимает заём по приглашению
•	Видит график платежей
•	Получает уведомления
•	Отмечает факт оплаты
•	Видит историю и статус займа
________________________________________
4. Основные пользовательские сценарии
⚠️ В сценарий подтверждения займа добавляется двухэтапное подтверждение с формированием и акцептом расписки между физическими лицами.
4.1 Создание и принятие займа
Со стороны мастера:
1.	Создает заём, указывая:
o	сумму
o	срок (в месяцах)
o	процентную ставку (годовую)
o	дату первого платежа
o	контакт заемщика (email или телефон)
2.	Система:
o	рассчитывает аннуитет и график
o	присваивает статус «Ожидает принятия»
o	создает уникальную ссылку приглашения
o	отправляет приглашение заемщику
Дополнительно (обязательно): данные кредитора
В мастер-аккаунте должен быть раздел «Данные кредитора», где указываются:
•	ФИО
•	Паспорт (серия, номер, кем и когда выдан)
•	Адрес регистрации
•	Реквизиты для перевода денежных средств (текстовое поле)
Эти данные используются исключительно для формирования предзаполненной расписки и не редактируются заемщиком.
________________________________________
Со стороны заемщика (двухэтапное подтверждение):
Этап 1 — подтверждение условий займа
1.	Переходит по ссылке
2.	Видит условия займа (сумма, срок, ставка, график)
3.	Создает аккаунт или логинится
4.	Ставит чекбокс:
«Я подтверждаю условия займа»
5.	Нажимает кнопку «Продолжить»
Статус займа остается: «Ожидает принятия»
________________________________________
Этап 2 — подписание расписки (конклюдентное действие)
6.	Заемщику отображается предзаполненная расписка в получении займа у физического лица, содержащая:
o	данные кредитора (ФИО, паспорт, адрес)
o	данные займа (сумма цифрами и прописью, срок, ставка)
o	обязательство вернуть сумму займа и проценты
7.	Заемщик заполняет свои данные:
o	ФИО
o	Паспортные данные
o	Адрес проживания
8.	Заемщик выполняет конклюдентное действие:
o	ставит чекбокс:
«Я подтверждаю, что получил(а) денежные средства и обязуюсь вернуть их на указанных условиях»
9.	Нажимает кнопку «Подписать расписку и принять заём»
После этого:
•	фиксируется дата и время подтверждения
•	сохраняется версия расписки (PDF / HTML)
•	заём получает статус «Активен»
•	включаются уведомления
•	мастер получает уведомление о принятии займа
________________________________________
5. Статусы займа
•	Черновик
•	Ожидает принятия
•	Активен (после подтверждения условий и подписания расписки)
•	Закрыт
•	Отменен
Признак «Просрочен» вычисляется динамически.
•	Черновик
•	Ожидает принятия
•	Активен
•	Закрыт
•	Отменен
Признак «Просрочен» вычисляется динамически.
________________________________________
6. Логика платежей
6.1 Платеж от заемщика
•	Заемщик отмечает платеж и вводит сумму
•	Платеж получает статус «Ожидает подтверждения»
•	До подтверждения расчеты не меняются
6.2 Подтверждение мастером
•	При подтверждении:
o	платеж становится «Подтвержденным»
o	применяется к расчетам
•	При отклонении:
o	статус «Отклонен»
o	график не меняется
________________________________________
7. Логика расчетов
7.1 Аннуитет
•	Расчет производится на сервере
•	График фиксируется при создании займа
7.2 Переплата
Если подтвержденный платеж больше аннуитета:
•	текущий период закрывается
•	излишек уменьшает тело долга
•	график пересчитывается на оставшийся срок
•	предыдущий график сохраняется для истории
7.3 Частичная оплата
•	Период не считается закрытым
•	Может привести к просрочке
________________________________________
8. Просрочка
•	Если текущая дата превышает дату платежа и период не оплачен — период считается просроченным
•	Уведомления отправляются мастеру и заемщику
•	Штрафы и пени в MVP не используются
________________________________________
9. Закрытие займа
Займ закрывается автоматически, если:
•	остаток долга равен 0
•	все периоды оплачены
________________________________________
10. Учет средств мастера
Для каждого подтвержденного платежа мастер может указать:
•	категорию использования средств:

o	расходы
o	резерв
o	инвестиции
•	комментарий

•	Подписание расписки реализуется как конклюдентное действие (чекбокс + кнопка), без ЭЦП в MVP
•	Сервер — единственный источник истины

22. Prisma Schema (MVP)
Ниже приведена базовая схема Prisma, соответствующая архитектуре БД MVP. Может использоваться напрямую в разработке.
model User {
  id        String   @id @default(uuid())
  role      Role
  email     String?  @unique
  phone     String?  @unique
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  creditorProfile CreditorProfile?
  borrowerProfile BorrowerProfile?
  loansAsMaster   Loan[] @relation("MasterLoans")
  loansAsBorrower Loan[] @relation("BorrowerLoans")
}

model CreditorProfile {
  id                  String   @id @default(uuid())
  userId              String   @unique
  fullName            String
  passportSeries      String
  passportNumber      String
  passportIssuedBy    String
  passportIssuedAt    DateTime

24. Sequence Diagrams (ключевые пользовательские потоки)
Ниже описаны последовательности взаимодействий между участниками системы. Формат — текстовый, но напрямую переводится в UML / Mermaid.
________________________________________
24.1 Поток №1: Создание и принятие займа
Участники:
•	Master (Кредитор)
•	PWA Frontend
•	Backend API
•	Borrower (Заемщик)
Последовательность:
1.	Master → Frontend: открывает «Создать заём»
2.	Frontend → Backend: запрос на расчет аннуитета
3.	Backend → Frontend: возвращает расчет и график
4.	Master → Frontend: подтверждает создание займа
5.	Frontend → Backend: POST /loans (status=PENDING)
6.	Backend → Frontend: loanId + invitationToken
7.	Backend → Notification Service: отправка ссылки заемщику
8.	Borrower → Frontend: переходит по ссылке
9.	Frontend → Backend: GET /loan-by-token
10.	Backend → Frontend: данные займа (read-only)
________________________________________
24.2 Поток №2: Подтверждение условий и подписание расписки
Участники:
•	Borrower
•	PWA Frontend
•	Backend API
Этап 1 — подтверждение условий:
1.	Borrower → Frontend: нажимает «Подтвердить условия»
2.	Frontend → Backend: POST /loan/{id}/accept-terms
3.	Backend:
o	фиксирует timestamp
o	сохраняет IP / User-Agent
4.	Backend → Frontend: OK
Этап 2 — подписание расписки:
5. Borrower → Frontend: вводит паспортные данные
6. Frontend → Backend: POST /loan/{id}/sign-receipt
7. Backend:
•	формирует snapshot сторон и займа
•	генерирует HTML расписки
•	сохраняет Receipt
•	меняет статус займа на ACTIVE
8.	Backend → Frontend: OK + loan active
9.	Backend → Notification Service: уведомление мастеру
________________________________________
24.3 Поток №3: Платеж, подтверждение и перерасчет
Участники:
•	Borrower
•	Master
•	PWA Frontend
•	Backend API
Отметка платежа заемщиком:
1.	Borrower → Frontend: вводит сумму платежа
2.	Frontend → Backend: POST /payments (status=DECLARED)
3.	Backend → Master: уведомление о платеже
Подтверждение мастером:
4. Master → Frontend: нажимает «Подтвердить платеж»
5. Frontend → Backend: POST /payments/{id}/confirm
6. Backend:
•	фиксирует confirmedAt
•	обновляет статус платежа
•	применяет платеж к графику
Перерасчет:
7. Backend:
•	если сумма > планового платежа:
o	пересчитывает остаток
o	генерирует новый график (новые записи)
8.	Backend → Frontend (Borrower): обновленный график
9.	Backend → Frontend (Master): обновленный статус займа
________________________________________
25. Примечания к sequence
•	Все критичные действия — POST
•	Повторы запросов должны быть idempotent
•	Все шаги логируются
•	Любая ошибка возвращает понятное состояние

1. Общая идея проекта
PWA-приложение для учета частных займов между физическими лицами.
